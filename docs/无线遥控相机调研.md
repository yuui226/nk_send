# 无线遥控相机(监看 + 曝光控制)可行性调研

> 2026-07-10 调研。目标:在现有 WiFi PTP/IP 连接上增加类似 SnapBridge「遥控拍摄」的功能——实时监看(Live View)、调整快门/光圈/ISO、遥控快门。

## 一、总结论:可行,且不需要换连接方式

现在 App 用的这条连接(相机 AP 模式,`192.168.1.1:15740`,PTP/IP)**就是遥控通道本身**。SnapBridge 的遥控拍摄、Cascable Studio、qDslrDashboard、ZineControl 等第三方 App 全部走的这条链路,用的是 Nikon 厂商扩展操作码。不需要 BLE 配对、不需要「连接到电脑(WT)」模式——相机菜单保持「连接到智能设备」即可。

需要在现有 `NikonCamera.kt` 之上新增三块:

1. **Live View**:厂商操作码启停 + 轮询拉 JPEG 帧
2. **曝光控制**:标准 `GetDevicePropDesc / GetDevicePropValue / SetDevicePropValue`(0x1014/0x1015/0x1016)读写快门/光圈/ISO 等属性
3. **遥控拍摄 + 事件**:厂商拍摄操作码 + 轮询 `GetEvent` 拿 ObjectAdded/CaptureComplete

## 二、最大风险:Z8/Z9 世代在智能设备模式下锁了协议

这是整个调研中唯一的硬钉子,直接决定实施策略:

- **Z5/Z50/Zfc/Z30/Z6/Z7(含二代)**:在「连接到智能设备」WiFi 下完整暴露遥控操作码集,qDslrDashboard 2018 年起就能对 Z6/Z7 改参数、拍摄、Live View。**直接照协议实现即可。**
- **Z8(fw 3.00)/Z9 世代(含 Zf/Z6III 等新固件机型,待验证)**:[libgphoto2 issue #1135](https://github.com/gphoto/libgphoto2/issues/1135) 实测,智能设备模式下 DeviceInfo 只广告标准操作码 + SnapBridge 世代的新厂商码(`0x935A`、`0x9439–0x9442`、`0x952B`),**不广告** `0x9201/0x9203/0x9207` 等遥控码;更狠的是 21 个标准属性虽然列在 DeviceInfo 里,但 `GetDevicePropValue` 一律返回 `0x2005 OperationNotSupported`。SnapBridge 显然先走了一套私有握手(`0x9439 GetVendorCodes` / `0x94xx` 认证族)才解锁。**开源界没有公开解法**;但 Cascable、ZineControl 等商业 App 在 Z8/Z9 上做到了遥控,证明纯协议层可解,需要自己抓包 SnapBridge 逆向。
- **实施策略**:OpenSession 后解析 GetDeviceInfo 的 OperationsSupported,按机型实际广告的操作码逐个 gate 功能。v1 先支持老世代机身,Z8/Z9 显示「暂不支持遥控」;后续抓包(Android 上可以对 SnapBridge 抓 pcap)再解锁新世代。

## 三、协议细节(来源:libgphoto2 ptp.h/library.c/config.c,已从源码核实)

### 3.1 Live View 操作码

| 操作码 | 名称 | 说明 |
|---|---|---|
| `0x9201` | StartLiveView | 无参数无数据 |
| `0x9202` | EndLiveView | 无参数无数据 |
| `0x9203` | GetLiveViewImg | 数据阶段 = 头部 + JPEG,轮询取帧 |
| `0x9428` | GetLiveViewImageEx | Z8/Z9 新增,可选;gphoto 至今仍对全系用 0x9203 |
| `0x90C8` | DeviceReady | 忙返回 `0x2019`,就绪返回 `0x2001`;StartLiveView 后要轮询它 |

**帧载荷格式**:固定长度元数据头 + 原始 JPEG(`FF D8 … FF D9`)。头长按世代不同:D7000 起到 Z5/Z6/Z6II/Z7/Z7II/Z50/Zfc 均为 **384 字节**;Z9 世代为 **1024 字节**(Nikon SDK Type0030)。**稳妥做法(gphoto 的做法)**:先按 384 校验该偏移处是否 `FF D8`,否则全量扫 SOI 标记定位 JPEG,不写死偏移。头部含 AF 框、旋转、对焦状态等元数据,Z 系无公开结构定义。

**Z 30 `0x9428` 实抓补充（2026-07-23）**：`focus_tap_trace.pcap` 中 82 帧的 JPEG 均从偏移 **512** 开始；头部为大端，版本前缀为 `00 01 00 00`，`+8` 回报头长、`+12` 回报 JPEG 长度、`+16/+18` 为完整坐标系。三次点按 AF 的终态依次为失败/失败/成功，`+42` 同步为 `1/1/2`（初始为 0）；成功帧 `+44/+45` 给出 AF 框数量/选中索引，`+48` 起为 `{宽,高,中心X,中心Y}`。正式实现只对完整通过版本、长度、JPEG 边界及坐标范围校验的 **单框** 512-byte 头启用 AF 框解析；多框或未知头型保留对焦状态，但框位稳定退回应用已知的点按坐标。增强帧除明确返回“不支持”外，连续两次异常才降级为 `0x9203`，避免偶发坏首帧导致整次连接永久丢失元数据。

**分辨率**:属性 `0xD1AC` LiveViewImageSize(u8:1=QVGA 320×240,2=VGA 640×480,3=XGA 1024×768)。帧约 30–100KB,拉取式轮询,WiFi 实测约 **5–15 fps**(单频段老机型偏低,Z6III 的 5GHz 明显更好),别按 30fps 设计 UI 预期。

**Live View 序列(照抄 gphoto 实现)**:
1. 读 `0xD1A2` LiveViewStatus,已开则跳过
2. 支持则设 `0xD10B` RecordingMedia=1;读 `0xD1A4` LiveViewProhibitCondition(u32 位掩码),非 0 则中止(bit24 镜头收纳中可忽略;bit31=模式不在 PASM,bit17=过热,bit15=正在拍摄…)
3. `StartLiveView`,容忍 `0x2019` DeviceBusy
4. 每 20ms 轮询 `DeviceReady` 直到就绪(上限 2s)
5. 循环 `GetLiveViewImg`;收 `0x2019` 等 10ms 重试;收 `0xA00B` NotLiveView 则静默重启 LV;**丢弃第一帧**(部分机身首帧损坏)

### 3.2 曝光控制属性(全部走标准 0x1014/0x1015/0x1016)

| 属性 | 代码 | 类型 | 编码 |
|---|---|---|---|
| 光圈 | `0x5007` FNumber | u16 | 光圈值×100(f/2.8=280);仅 A/M 可写,P/S 下返回 `0xA00A` |
| 快门 | `0xD100` Nikon ExposureTime | u32 | **高16位分子 \| 低16位分母**:1/250s=`0x000100FA`,1.5s=`0x000F000A`;`0xFFFFFFFF`=Bulb,`0xFFFFFFFD`=Time;仅 S/M 可写 |
| 快门(标准) | `0x500D` | u32 | 秒×10000,表达不了 Bulb,Nikon 工具都用 0xD100 |
| ISO | `0x500F` ExposureIndex | u16 | 手动/基础 ISO；超 65535 的扩展 ISO 用 `0xD0B4`(u32)；自动 ISO 的实时结果读取只读 `0xD0B5`(u32) |
| 曝光补偿 | `0x5010` | i16 | 千分之一 EV:+0.3EV=300 |
| 曝光模式 | `0x500E` | u16 | 1=M,2=P,3=A,4=S;**带物理模式拨盘的机身(Z8/Z9 以外全部)只读** |
| 白平衡 | `0x5005` | u16 | 标准值 + Nikon `0x8010+` 扩展 |

**关键**:用 `GetDevicePropDesc` 的 GetSet 标志 + 枚举 FORM 决定 UI 可写性和可选值列表——相机会按当前曝光模式动态重新枚举;不可写时直接 SetDevicePropValue 会被拒。另外 Nikon 的 `0xD1xx` 属性经常**不出现在 DeviceInfo 的属性列表里**,要用 `0x90CA GetVendorPropCodes`(新机型 `0x9439 GetVendorCodes`,参数 0x0D)显式查询。

### 3.3 遥控拍摄

| 操作码 | 名称 | 说明 |
|---|---|---|
| `0x9207` | InitiateCaptureRecInMedia | **现代机身首选**。p1=`0xFFFFFFFE`(先AF)/`0xFFFFFFFF`(不AF);p2=0 存卡 / 1 存SDRAM |
| `0x90C0` | InitiateCaptureRecInSdram | 老式,存 SDRAM |
| `0x90CB` | AfCaptureSDRAM | AF 后拍摄到 SDRAM |
| `0x920C` | TerminateCapture | 结束 Bulb,参数 (0,0) |
| `0x90C1` / `0x9206` | AfDrive / AfDriveCancel | AfDrive 只启动 AF，最终结果由 DeviceReady 返回：`0x2001` 合焦，`0xA002` OutOfFocus |
| `0x9204` / `0x9205` | MfDrive / ChangeAfArea | 手动对焦驱动 / 触摸对焦点 (x,y) |
| `0x920A` / `0x920B` | 录像开始 / 结束 | Z 系需先设属性 `0xD1F0`=1 + `0x9435 ChangeApplicationMode(1)`,并检查 `0xD0A4` MovRecProhibitCondition==0 |

**拍摄序列**:发 0x9207(对 `0x2019/0xA004/0xA207` 重试)→ 轮询 DeviceReady(100ms 间隔,长曝光可能很久)→ 轮询 GetEvent 直到同时看到 ObjectAdded 和 CaptureComplete → GetObjectInfo/GetObject 取图(SDRAM 图常用伪句柄 `0xFFFF0001`)。LV 开启时拍摄须用不 AF 变体,拍完 gphoto 会重启 LV。

**点按对焦序列（Z 30 实抓）**：用当前 Live View JPEG 像素坐标发
`0x9205 ChangeAfArea(x,y)` → 等待约 80ms（三次实抓为 80–84ms）→
只发一次 `0x90C1 AfDrive` → 轮询
`0x90C8 DeviceReady`。必须等待 DeviceReady 的终态，不能把 AfDrive 自身的
OK 当成合焦成功。

### 3.4 事件

Nikon 在 WiFi 下**不靠 PTP/IP 事件通道**发拍摄事件(现有代码把事件包直接忽略是对的,通道保持连着当 keep-alive 即可),正规机制是命令通道上轮询:

- `0x90C7 GetEvent`:载荷 = `u16 count` + count × `{u16 eventCode, u32 param1}`(小端,每条 6 字节)
- 新 Z 机型广告 `0x941C GetEventEx`(多参数事件)时优先用它
- 关心的事件码:`0x4002` ObjectAdded、`0x400D` CaptureComplete、`0x4006` DevicePropChanged、`0xC101` ObjectAddedInSDRAM、`0xC102` CaptureCompleteRecInSdram、`0xC10C` LiveViewStateChanged、`0xC10A/0xC108/0xC105` 录像开始/完成/中断

### 3.5 需要处理的 Nikon 响应码

`0x2019` DeviceBusy(重试)、`0xA002` OutOfFocus、`0xA004` InvalidStatus(如 Z8 处于上传优先模式时拒绝 LV)、`0xA00A` 当前模式不可调光圈、`0xA00B` NotLiveView(重启 LV)、`0xA008` ShutterSpeedBulb、`0xA200/0xA201` Bulb/静音释放忙(wait-busy 循环里当 OK)。另:部分机身 LV 连续 ~14000 帧后自动掉出并返回 `0xA00B`,gphoto 静默重启。

## 四、实际体验限制(设计 UI 时要考虑)

1. **遥控会话期间机身被锁**:相机屏幕显示「已连接到智能设备」,机身按键/取景器不可操作(SnapBridge 遥控同样如此)。所以「传输」和「遥控」必须是两个显式模式,像 SnapBridge 一样切换,不能默默常驻遥控态。
2. **传输与 Live View 互斥于同一 PTP 会话**:PTP 事务严格串行,一个 50MB NEF 的 GetObject 期间 LV 会停摆。方案:要么模式切换(SnapBridge 做法),要么下载改用 GetPartialObject 分片,在帧间隙穿插(注意热路径零开销约束——分片只在遥控模式下用,普通传输路径不动)。
3. **曝光模式门控**:SnapBridge 遥控要求机身在 P/S/A/M;光圈仅 A/M 可写、快门仅 S/M 可写;模式拨盘机身无法远程切换曝光模式(Z8/Z9 可以)。UI 按 GetDevicePropDesc 结果置灰而不是报错。
4. **单客户端**:相机 AP 同时只接受一个控制会话(SnapBridge 与 NX MobileAir 也互斥)。
5. **SnapBridge 自己的已知短板**(可作差异化卖点):遥控时不能连拍、RAW 拍了不自动回传、自拍定时限 2s——第三方做得更好是有市场空间的。

## 六、建议实施路线

1. **阶段 0(半天)**:在现有连接上发 GetDeviceInfo,dump OperationsSupported/DevicePropertiesSupported + `0x90CA` 结果,用手头真机确认操作码集(尤其确认自己机器属于哪个世代)。
2. **阶段 1:Live View 监看**(核心体验):StartLiveView 序列 + 轮询取帧解 JPEG 显示 + EndLiveView;遥控模式作为独立页面/模式与传输互斥。
3. **阶段 2:曝光控制**:GetDevicePropDesc 驱动的快门/光圈/ISO/曝光补偿选择器(枚举值来自相机,不硬编码档位表)。
4. **阶段 3:遥控快门**:0x9207 + GetEvent 轮询 + 拍后小图回显(可复用现有缩略图管线)。
5. **阶段 4(可选)**:触摸对焦(ChangeAfArea)、录像、Bulb、Z8/Z9 世代解锁(抓 SnapBridge 包逆向 0x94xx 握手)、BLE 自动唤醒。
