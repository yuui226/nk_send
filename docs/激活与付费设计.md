# 激活与付费体系设计

> 状态:设计稿,待确认后实施。基线:`paid-prep` 标签(master)。
> 模式:免费版(未激活)+ 付费激活;服务器激活,本地通行证离线可用;单设备浮动授权(换机顶替旧机)。

---

## 1. 授权模型

| | 免费版 | 付费版 |
|---|---|---|
| 获取方式 | 直接安装即用 | 购买激活码,联网激活一次 |
| 授权载体 | 无 | 本地通行证(服务器 ECDSA P-256 签名) |
| 联网要求 | 无 | 仅激活/换绑时需要;日常完全离线 |
| 设备数 | — | 一码最多 2 台设备 |

同一个 APK 同时承载两种形态,靠激活状态切换,不出两个包。

## 2. 免费版功能限制清单(待确认)

原则:免费版必须完整证明"能连上我的相机、真的能传出来"(这是转化的前提),
但对职业化使用场景(批量、RAW、遥控)设卡。

| # | 功能 | 免费版 | 付费版 | 理由 |
|---|------|--------|--------|------|
| 1 | 连接相机 / 浏览文件列表 / 缩略图 | ✅ 完整 | ✅ | 必须让用户先看到自己的照片,证明兼容性 |
| 2 | 大图预览(含缩放) | ✅ 完整 | ✅ | 看得到传不走,是最好的付费动机 |
| 3 | 传输(照片/视频/RAW 统一计数) | ⚠️ 每天 25 个,入队即计数 | ✅ 无限 | 核心限制。所有格式共享同一额度,够验证不够干活 |
| 5 | 整组传输 / 全部加入队列 | ❌ 只能单张点传 | ✅ | 批量是效率工具的核心价值 |
| 6 | 监看 / 遥控(live view、调参、录像) | ⚠️ 每次进入限 2 分钟体验 | ✅ 无限 | 让用户尝到,但没法拿去干活 |
| 7 | 传输队列管理(重试/清空等) | ✅ | ✅ | 跟随 #3/#5 的额度,无需单独限制 |
| 8 | 设置(目录/外观/语言等) | ✅ 完整 | ✅ | 基础体验不设卡 |

补充规则:

- 计数规则:任务**加入队列即 +1**(不论后续成功/失败/取消);同一任务点"重试"
  不重复计;从队列移除后重新加入则重新计。每天 25 个,按自然日重置,存本地;
  用户改系统时间能绕过——**接受**,防君子不防小人,别为它加复杂度。
- 免费版触碰限制时,统一弹同一个付费引导页(说明差异 + QQ 群购买方式 + 输入激活码入口)。
- 限制数值(25 个 / 2 分钟)做成常量集中定义,便于日后调整。

## 3. 客户端设计

### 3.1 设备指纹

```
fp = SHA-256(ANDROID_ID + applicationId) 取前 16 字节,hex
展示形式:ZT-XXXX-XXXX(取 fp 前 8 字符分组,仅用于人读/客服沟通)
```

- `ANDROID_ID` 在恢复出厂 / 换签名 key 时会变 → 视为新设备,占用新槽位,换绑解决。
- 服务器通信中始终用完整 fp,展示码仅用于人工场景。

### 3.2 激活状态机

```
FREE(未激活)──输入激活码,联网激活成功──▶ PRO(已激活)
PRO ──续签时服务器返回"已吊销/已解绑"──▶ FREE(清除本地通行证)
PRO ──通行证本地验签失败/指纹不符──▶ FREE
断网 / 服务器不可达:状态不变(离线不惩罚)
```

### 3.3 通行证(License Token)

服务器签发,App 本地存储 + 每次启动验签:

```
payload(JSON,UTF-8):
{
  "v": 1,                     // 格式版本
  "code": "KRMXTP",           // 激活码(6 位大写字母)
  "fp": "<完整设备指纹>",
  "plan": "pro",              // 预留分级
  "iat": 1752480000           // 签发时间(unix 秒)
}
token = base64url(payload) + "." + base64url(ECDSA_P256_SHA256.sign(私钥, payload))
```

- 签名算法用 **ECDSA P-256(SHA256withECDSA)** 而非 Ed25519:Android JCA 的 Ed25519
  需要 API 33,P-256 在 minSdk 26 上原生可用,双端零第三方依赖。
- App 内置 P-256 **公钥**,验签 + 核对 fp == 本机指纹,通过即 PRO。
- 通行证**不设过期时间**,离线永久有效;吊销依赖续签(3.4)。
- 存储:`SharedPreferences`(私有目录,无需加密——伪造需要私钥,复制到别机指纹不符)。

### 3.4 静默续签

- 触发:App 启动且距上次成功续签 ≥ 7 天,且当前有网;后台静默,失败无感。
- 成功:覆盖本地通行证,刷新 `last_renew_at`。
- 服务器明确返回 `CODE_REVOKED` / `NOT_BOUND`:清除通行证,降级 FREE,弹提示。
- 网络失败 / 服务器宕机 / 其他错误:什么都不做。

### 3.5 服务器地址与"提示更新"

```kotlin
// 内置地址列表,按序尝试;将来有域名/换 IP 时在新版里更新此列表
val SERVERS = listOf("https://<ECS公网IP>:8443")
```

- 仅在**用户主动操作**(激活、手动刷新)时,所有地址全部失败 →
  提示:"无法连接激活服务器,请检查网络;若网络正常,可能需要更新到新版本(QQ 群获取)"。
- 静默续签失败不弹任何提示。

### 3.6 传输层安全

- 自签名证书(有效期 10 年),App 内做 **certificate pinning**(只信任这一张证书),
  不开 `usesCleartextTraffic`。
- 证书公钥指纹作为常量编译进 App。

### 3.7 UI 落点

- 设置页:新增"激活"条目(FREE 显示"免费版 · 点击激活",PRO 显示"已激活 · 设备码 ZT-XXXX-XXXX")。
- **测试阶段**:设置页 PRO 状态下额外显示"恢复为免费版"按钮(清除本地通行证,
  不动服务器绑定,重新输入激活码即可恢复 PRO)。公开发售前移除或藏入开发者面板。
- 激活页:显示本机设备码、激活码输入框、激活按钮、隐私说明一行
  ("激活需将设备标识发送至激活服务器,仅用于授权验证")。
- 付费引导页(触限时弹出):免费/付费差异表 + QQ 群号 + "输入激活码"按钮。

## 4. 服务端设计

### 4.1 形态

- 单二进制小服务(建议 Go;备选 Node/Python 均可),systemd 常驻,监听 `:8443`(TLS,自签证书)。
- 数据:SQLite 单文件(每日 cron 备份即可)。
- 私钥(ECDSA P-256 签发用)只存服务器,权限 600;管理接口用长随机 `X-Admin-Token` 鉴权。
- 2 核 2G 远超需求,无需缓存/队列/容器。

### 4.2 接口总览

公开接口(App 调用):

| 方法 | 路径 | 用途 |
|------|------|------|
| POST | `/v1/activate` | 激活:绑定设备并签发通行证 |
| POST | `/v1/renew` | 续签:校验绑定仍有效,重发通行证 |

管理接口(仅你使用,需 `X-Admin-Token` 头):

| 方法 | 路径 | 用途 |
|------|------|------|
| POST | `/admin/codes` | 生成激活码 |
| GET | `/admin/codes` | 台账:列出码及绑定情况 |
| POST | `/admin/unbind` | 解绑某码上的某设备(换机) |
| POST | `/admin/revoke` | 吊销整个码 |

### 4.3 接口定义

统一约定:请求/响应均 JSON;业务失败返回 HTTP 200 + `{"ok":false,"err":"<错误码>"}`,
便于客户端统一处理;仅鉴权失败(管理接口)返回 401。

#### POST /v1/activate

```json
请求: {
  "code":  "KRMXTP",
  "fp":    "<完整设备指纹 hex>",
  "model": "Pixel 8",          // 可选,仅用于台账展示
  "app_ver": "1.0"             // 可选,排障用
}
成功: { "ok": true, "token": "<base64url.payload>.<base64url.sig>" }
失败: { "ok": false, "err": "CODE_NOT_FOUND | CODE_REVOKED | SLOTS_FULL" }
```

逻辑:码存在且 `status=active` → 查绑定:该 fp 已绑定则直接重发通行证(幂等,
支持卸载重装);未绑定且槽位 < `max_devices` → 插入绑定、签发;槽满 → `SLOTS_FULL`。

#### POST /v1/renew

```json
请求: { "code": "...", "fp": "...", "app_ver": "1.0" }
成功: { "ok": true, "token": "<新通行证>" }
失败: { "ok": false, "err": "CODE_REVOKED | NOT_BOUND" }
```

逻辑:码 active 且 (code, fp) 绑定存在 → 更新 `last_renew_at`,重签发;
否则返回对应错误(客户端据此降级)。

#### POST /admin/codes

```json
请求: { "count": 5, "note": "7月QQ群批次", "max_devices": 2 }
成功: { "ok": true, "codes": ["KRMXTP", ...] }
```

码格式:6 位纯大写字母,去除易混淆的 `I O`(字符集 24 字母)。空间 24^6 ≈ 1.9 亿,
配合 `/v1/activate` IP 限速足以挡枚举(手动发码的小众场景)。服务器生成保证唯一,
撞码自动重试。设备码仍带 `ZT-` 前缀(见 3.1),与激活码格式不同,便于用户区分两者。

#### GET /admin/codes

```json
响应: { "ok": true, "codes": [ {
  "code": "KRMXTP", "status": "active", "note": "7月QQ群批次",
  "created_at": "2026-07-14T10:00:00Z",
  "bindings": [ { "fp": "...", "model": "Pixel 8",
                  "activated_at": "...", "last_renew_at": "..." } ]
} ] }
```

#### POST /admin/unbind

```json
请求: { "code": "ZT-...", "fp": "<要解绑的指纹,或其 ZT-XXXX-XXXX 展示码前缀>" }
成功: { "ok": true }
```

#### POST /admin/revoke

```json
请求: { "code": "ZT-...", "reason": "退款" }
成功: { "ok": true }
```

### 4.4 数据库表结构(SQLite)

```sql
CREATE TABLE codes (
  code        TEXT PRIMARY KEY,             -- 6 位大写字母,如 KRMXTP
  status      TEXT NOT NULL DEFAULT 'active', -- active | revoked
  max_devices INTEGER NOT NULL DEFAULT 2,
  note        TEXT,                          -- 批次/买家备注
  created_at  TEXT NOT NULL,                 -- ISO8601 UTC
  revoked_at  TEXT,
  revoke_reason TEXT
);

CREATE TABLE bindings (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  code          TEXT NOT NULL REFERENCES codes(code),
  device_fp     TEXT NOT NULL,
  device_model  TEXT,
  app_ver       TEXT,
  activated_at  TEXT NOT NULL,
  last_renew_at TEXT,
  UNIQUE (code, device_fp)
);
CREATE INDEX idx_bindings_code ON bindings(code);
```

解绑 = 物理删除 bindings 行(台账需要留痕的话后续加 audit 表,初期不做)。

### 4.5 安全要点

- 私钥、admin token 不入库不入 git;部署时手工放服务器。
- `/admin/*` 只接受带正确 token 的请求;可再加阿里云安全组白名单(仅你家 IP)双保险。
- `/v1/*` 无需防重放:activate 幂等、renew 无副作用,重放无利可图。
- 频率限制:同一 IP 对 `/v1/activate` 简单限速(如 10 次/分钟),防激活码爆破
  (码空间 32^12,爆破本不可行,限速是加保险)。
