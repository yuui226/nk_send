# 连接到手机热点(相机「连接到电脑」模式)可行性调研

> 2026-07-13 调研。起因:用户反馈 Z50 有「连接到电脑」菜单——相机加入 Wi-Fi 后由电脑端扫描局域网发现并传输,询问手机 App 能否实现同样模式。
> **方向存疑,待实机验证后再定**(见第一节更正结论):"手机开热点、相机来连"在主力机型 Z30/Z50 上恰好落在最难的一条路上。

## ⚠ 更正说明(2026-07-13 二次调研)

初版把配对握手写成「唯一硬工作量、抓包复刻报文即可」。二次求证(gphoto2/airnef 源码与 issue、官方手册、ShutterSnitch 论坛一手实证)后**推翻了这个判断**,要点更正如下,细节见第一节:

1. 配对**真实存在**(官方强制、用户实证不配对连不上),但它**不是一段可复刻的报文握手**——认证码大概率是**带外**的,相机把配对过的 host 身份记在自己那儿,抓包也看不到"认证码字段"。
2. 真正挡路的是 STA 模式下**相机只开放下载类操作码 + 会话被相机主动掐断**,且**开源界至今无人跑通**老机型的完整「连接到电脑」流程。
3. **世代分水岭**:老机型(Z30/Z50/Zfc/Z6/Z7/Z6II/Z7II)只能走「连接到电脑」+WTU 配对才能加入外部热点;新机型(Z50II/Z6III/Z8/Z9/Zf)的「连接到智能设备」新增 STA 模式,免配对。

## 一、总结论(更正版):握手真实但性质不同,主力机型落在最难路径

- 协议同源没变:「连接到电脑」底层就是 **PTP/IP(TCP 15740)+ mDNS 发现(UDP 5353)**,与现有「智能设备」连接(相机 AP,`192.168.1.1:15740`)同协议;**连接一旦建立,浏览/缩略图/下载/传输逻辑 100% 复用,一行不改**。问题全在"如何把连接建起来"。
- **配对是真实关卡,但不是我们想的那种**:
  - 存在性【确证】:Z8/Z9/Z50/Z30 官方手册明写"连接前必须用 WTU 配对,相机显示认证码输入电脑";ShutterSnitch 用户【一手实证】"退出 WTU、把相机 IP 填进第三方 App 就连不上"。
  - 性质【强证据】:GUID **不是**严格关卡——airnef 用**写死的假 GUID**、gphoto2 用自带随机 GUID 都能完成 Init+OpenSession(RC=0x2001)。认证码**未进 PTP/IP 报文**,是带外步骤。→ **抓包也复刻不出一个"认证码握手",相机是按记住的 host 身份放行。**
- **真正的障碍(gphoto2 #1135,连 Z8 维护者亲测)**:
  1. STA/PC 模式下相机**只吐下载类操作码**(对纯传输 App 反而够用);
  2. 相机 **Init 后主动关会话**,除非客户端保持会话——这是真正待攻克的工程点;
  3. **开源界尚无人公开跑通**老机型完整「连接到电脑」流程,#1135 仍卡住。
- **世代分水岭(决定要不要走这条路)**:
  - **老机型 Z30/Z50/Zfc/Z6/Z7/Z6II/Z7II(我们的主力机)**:「连接到智能设备」**仅 AP 模式**(相机做热点),没有加入外部网络的选项。要加入手机热点**只能走「连接到电脑」+WTU 配对,绕不开**。
  - **新机型 Z50II/Z6III/Z8/Z9/Zf**:「连接到智能设备」**新增 STA 模式**,可直接加入手机热点,**免 WTU**。
- 其余连接层工作(不变):相机 IP 由手机 DHCP 分配、不固定,需 mDNS 或网段扫描定位(探针脚本已实现)。

## 一·B、三个选择(实机验证前的决策框架)

| 方案 | 做法 | 适用 | 代价/风险 |
|---|---|---|---|
| **A. 验证 PC 模式** | 抓包搞清 Z50 在 PC 模式下配对后认什么(GUID?)、会话怎么保持、放行哪些操作码 | 想要"手机热点"体验、且愿意啃会话保持 | 公共资料空白正在此,只能自测;可能卡在掐会话 |
| **B. 退回 AP 模式** | 沿用"相机做热点、手机连 192.168.1.1"(免配对、airnef 已证明可下载),用 `bindProcessToNetwork` 压住 Android 断网切网问题 | 老机型,追求稳、成本低 | 手机连相机热点时本机不上网(但 `bindProcessToNetwork` 后其他 App 仍走蜂窝) |
| **C. 按机型分档** | 新机型走智能设备 STA(免配对、干净);老机型退回 AP 模式 | 兼顾体验与覆盖 | 两套路径,UI/逻辑复杂度上升 |

> 选 B/C 时,`bindProcessToNetwork`(旧名 `setProcessDefaultNetwork`,API 23+)是关键:把 App 进程 socket 强制绑到相机那张无内网的 Wi-Fi 上,数据包不再被系统甩到蜂窝口。用 `NetworkRequest`(去掉 `NET_CAPABILITY_INTERNET`)+ `requestNetwork` 拿到 `Network` 后 `bindProcessToNetwork(network)`,或对单 socket `network.bindSocket()`。这也是现有 AP 模式就该做的稳定性加固。

**建议**:先以最小成本做第六节的抓包实验,但把它重定位成"探清 Z50 PC 模式的真实行为"而非"复刻握手";若配对后相机按 GUID 认且能保持会话 → 方案 A 成立;若掐会话啃不动 → 走方案 B。

## 二、机型支持情况(来源:WTU 官方支持列表)

**全部 Z 系列均支持内置 Wi-Fi「连接到电脑」**:Z9、Z8、Z6III、Z7II、Z6II、Z7、Z6、Z5II、Z5、Zf、Z50II、Z50、**Z30**、Zfc、ZR(另有 D780、D6)。没有机型必须外接 WT-7(它只是 Z5/Z6/Z7 两代的选配扩展件)。

- **老单反不支持**:D850/D750/D500/D7200 等的内置 Wi-Fi 不能与 WTU 配对,走不了这条路。
- 该菜单有两种子模式(Z50 手册明确):**接入点模式**(相机做热点)和 **基础结构/station 模式**(相机作为客户端加入已有 AP)。手机热点对相机而言就是普通 AP,可用 station 模式加入——**但注意:老机型只有「连接到电脑」菜单提供 station,「连接到智能设备」在老机型上仅 AP**(见第一节世代分水岭)。职业圈的"加入 iPhone 热点向 ShutterSnitch 传图"先例集中在 Z8/Z9 等新机型。
- 官方设计下该模式仅做图传(选片上传 / 新照片自动推送 Auto upload;视频只能手动选传);仅 Z8/Z9/Z6III 官方支持经网络遥控(NX Tether)。中端机型在此模式下协议层是否暴露遥控操作码未验证。
- 来源:
  - WTU 帮助(机型清单+配对流程):https://nikonimglib.com/wt4su/onlinehelp/en/02_pairing_01.html
  - Z30 手册:https://onlinemanual.nikonimglib.com/z30/en/10-02.html · 菜单页 https://onlinemanual.nikonimglib.com/z30/en/09-06-27.html
  - Z50 双模式说明:https://onlinemanual.nikonimglib.com/z50/en/10_establishing_wireless_connections_02.html
  - Nikon USA 连接方式总览:https://www.nikonusa.com/learn-and-explore/c/tips-and-techniques/z-series-wireless-connectivity-options

## 三、协议事实

- Z8/Z9 手册明确:连接电脑使用 **TCP 15740 + UDP 5353**(https://onlinemanual.nikonimglib.com/z8/en/computers_connecting_via_wireless_lan_90.html)。
- mDNS 服务名**大概率**是 `_ptp._tcp`(PTP/IP 生态通用),无尼康一手抓包证据,**待实机验证**。
- PTP/IP Init Command Request = 16 字节 GUID + UCS-2 Friendly Name + 版本号(权威参考:libgphoto2 `camlibs/ptp2/ptpip.c`)。
- **GUID 是否准入凭据?——证据更正**:早期资料称"需从 WTU 流量嗅探 GUID",但二次求证否定了 GUID 是严格关卡:airnef 用**硬编码假 GUID**(`airnefcmd.py` default `0x7766...9988`)、gphoto2 用自带随机 GUID(issue #1135 日志 GUID=`bd bb d8 81...`、name="INI-K")均能完成 Init+OpenSession(RC=0x2001)。GUID 只是"标识 host",客户端可自定义。
- **真正障碍来源**:gphoto2 #1135(连 Z8)——STA 模式相机"not offer any device property management opcodes... really just download options",且会话被相机主动关闭需保持。
- 证据来源:[gphoto2 #1135](https://github.com/gphoto/libgphoto2/issues/1135) · [airnef airnefcmd.py](https://github.com/besi/airnef/blob/master/airnefcmd.py) · [ShutterSnitch Z6II 帖(WTU 不可省的一手实证)](https://www.shuttersnitch.com/forums/viewtopic.php?f=5&t=2961)

## 四、为什么可能更快(以及要压住的预期)

速度优势不只来自频段,分三层:

1. **频段/PHY(主因)**:相机做软 AP 时固件通常锁 2.4GHz、单流、20MHz(省电 + 法规:5GHz DFS 信道不允许设备主动做 AP);做 station 时可协商 5GHz/11ac/80MHz。同一块芯片,换角色速率上限不同。
2. **对端射频质量**:手机/路由器的天线和功率优于相机软 AP,同距离信噪比更好。
3. **反向因素**:经路由器多一跳、空口时间翻倍(同信道砍半)——手机热点场景不存在这一条,相机↔手机仍是单跳,这点比「家用路由器 station」更优。

**预期管理**:很多机型瓶颈在相机 PTP 协议栈而非无线层——老机型无论信号多好约 2.5MB/s,Z9 走 5GHz 实测约 16MB/s 封顶(https://www.dpreview.com/forums/thread/4645624)。Z30/Z50 的实际提速幅度必须真机实测。**更确定的收益是手机不断网 + 连接稳定,速度是锦上添花。**

## 五、手机热点方案的 Android 侧约束

| 事项 | 结论 |
|---|---|
| 手机全程有网 | 热点共享蜂窝网络,彻底消灭「连相机热点后系统切网/断网」一整类问题;户外无路由器也可用 |
| App 代开热点 | **不行**(Android 8+ 禁止第三方开系统热点),只能引导用户手动开(可跳设置页)。`startLocalOnlyHotspot()` 的 SSID/密码每次随机,相机端要手动录入凭据 → 不可用。系统热点凭据固定,相机录一次即可自动重连 |
| 相机 IP 发现 | DHCP 分配不固定;NsdManager 在自家热点接口上跑 mDNS 历来不可靠 → 兜底方案:热点网段 /24 快扫 TCP 15740 |
| 加密兼容 | 新手机热点可能默认 WPA3,老相机可能仅 WPA2 → 引导用户设 WPA2/WPA3 兼容 |
| 配对 | 「连接到电脑」模式首次连接强制 WTU 配对;但配对**不是可复刻的报文握手**(认证码带外),真正难点是会话保持+操作码开放(见第一节) |

## 六、前置验证实验(重定位:探清 Z50 PC 模式的真实行为,而非"复刻握手")

初版把这步定为"抓包复刻认证码握手";更正后目标改为**用实测填补公共资料的空白**——搞清楚 Z50 在「连接到电脑」模式下的真实机制,据此在方案 A/B/C 间做决策。尼康官方唯一能完成配对的软件 WTU 只有 Win/Mac 版,故用 PC 做一次性实验(与产品无关,最终用户不需要电脑):

1. PC 装 WTU(下载页 https://downloadcenter.nikonimglib.com/en/download/sw/272.html),与 Z50/Z30 真机走一遍官方配对,全程 Wireshark 抓包;
2. **先跑探针脚本** `nikon_ptpip_probe.py`:看未配对时 Init 是 Ack 还是 Fail、fail_reason 是什么、能否 OpenSession——直接判定配对是不是硬门槛;
3. 抓包重点看(不再指望找"认证码字段"):
   - 配对**后**,相机放行连接靠的是什么?我方 GUID 是否必须与配对时一致(换 GUID 会不会被拒)→ 决定我们要不要持久化一个固定 GUID;
   - 会话建立后相机**多久掐断**、WTU 用什么(心跳/Ping 0x9F?保持某操作码轮询?)维持会话不掉;
   - 该模式 GetDeviceInfo 实际广告哪些 OperationsSupported(是否只有下载类);
4. mDNS 服务名与广播行为;相机在手机热点下的实际吞吐;Z30/Z50 station 能否上 5GHz。

**判定出口**:
- 若"配对后按固定 GUID 放行 + 会话可保持 + 下载操作码齐全" → **方案 A 成立**,复刻会话保持逻辑即可(非复刻握手);
- 若"会话保持啃不动 / 该模式连下载都受限" → **转方案 B**(AP 模式 + `bindProcessToNetwork`),App 主体逻辑不动。

## 七、实施路线(按验证结果分叉)

**若走方案 A(手机热点)**:
1. 连接页双模式 UI:「相机热点直连」(现状)/「相机连手机热点」;
2. 手机热点引导:跳系统设置开热点 → 展示 SSID/密码供用户录入相机 → 等待相机入网;
3. 发现:mDNS(NsdManager)优先,/24 端口扫描兜底(探针已实现);
4. 固定 GUID + 会话保持接入现有 PTP/IP 连接层(仅新模式路径,**不动现有热路径**);
5. 会话与传输逻辑全部复用现有代码。

**若走方案 B(AP 模式加固,成本最低)**:
1. 现有"相机做热点、手机连 192.168.1.1"路径加 `bindProcessToNetwork`,消除 Android 断网切网;
2. 传输逻辑零改动;连接页无需双模式(或仅作稳定性提示)。
