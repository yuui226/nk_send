# Z传（Nikon PTP/IP 传输）— 功能逻辑分析与缩略图方案

> 版本：截至缩略图功能开发前的主分支。本文按**使用场景**梳理全部功能逻辑，评估是否满足用户需求、是否合理、可优化点，并给出文件列表缩略图的最佳性能实现方案与本次落地设计。

---

## 一、整体架构

| 层 | 组件 | 职责 |
|---|---|---|
| 协议层 | `NikonCamera` / `PacketReader` / `PtpConstants` | PTP/IP 握手、命令收发、文件信息/下载/缩略图，全部经 `ioMutex` 串行化（单 socket） |
| 状态层 | `CameraViewModel` | 连接生命周期、Wi-Fi 检测/重连、文件列表加载、心跳、缩略图缓存 |
| 状态层 | `TransferViewModel` | 传输队列、并发下载状态、目录设置、显示设置 |
| 服务 | `TransferService` | 传输期间前台保活 + 唤醒锁 |
| UI | Compose 屏幕（Home/FileList/Transfer/Settings） | 展示与交互 |

**设计主线**：单条 PTP/IP 命令通道 + 独立事件通道；所有相机 IO 由 `ioMutex` 串行，避免并发读写同一 socket。

## 二、按使用场景的功能逻辑

### 场景 1：打开 App / 连接相机
**用户预期**：连上相机 Wi-Fi 后自动连接，不用手动操作。

**当前逻辑**：
- `registerNetworkCallback` 监听 Wi-Fi；`startConnectionWatcher` 每 3s 兜底检测。
- `isNikonWifi()` 读本地 DHCP 网关判断是否在相机网段（`192.168.1.1`）。
- 一旦在相机 Wi-Fi 上且未连接 → `connectToCameraWithRetry()` **持续重试直到连上**，不报错。
- 连上后自动跳转文件列表并开始加载。

**评估**：✅ 合理，覆盖"先开 App 后连 Wi-Fi""先连 Wi-Fi 后开 App"两种顺序。不在相机 Wi-Fi 上时只做极廉价的本地网关检测、不发无谓 socket，省电。

**可优化**：
- `isNikonWifi` 依赖 `dhcpInfo.gateway`（已废弃 API，个别机型可能返回 0）。可考虑改用 `ConnectivityManager.getLinkProperties()` 的路由/网关信息作为更稳的判断。
- 目前硬编码相机 IP `192.168.1.1`，仅适配默认网段；如需支持其它尼康机型/网段，可将其做成可配置或按已知网段列表匹配。

### 场景 2：浏览文件列表
**用户预期**：快速看到相机里的照片/视频，按日期分组。

**当前逻辑**：
- `getStorageIds` → `getObjectHandles` → `streamFileInfo` 分批（每批 20）拉取 `ObjectInfo`，边拉边显示。
- handle 降序 ≈ 时间由新到旧；最终展示由 `groupFilesByDate` 按日期分组、组内按拍摄时间倒序。
- 列表项显示：类型角标（JPG/NEF/MOV 配色）、文件名、大小、传输状态。

**评估**：✅ 分批流式加载体验好，先出内容不阻塞。分组逻辑清晰。

**可优化**：
- **无缩略图**——纯文本列表，浏览照片时无法凭内容识别，是当前最大的体验短板（本次要补）。
- `ObjectInfo` 的文件大小是 PTP 的 32 位 `ObjectCompressedSize`，>4GB 视频会截断（协议限制，尼康多数场景 OK）。
- 缺少筛选（只看 RAW/只看视频/只看某日）。文件很多时定位不便。

### 场景 3：传输文件
**用户预期**：点一下就传，能看进度/速度，传得快。

**当前逻辑**：
- 未设目录 → UI 引导到设置页；设了目录才允许入队。
- 队列 FIFO，按 `handle` 稳定键定位任务；单 socket 串行下载。
- 下载走**零拷贝读包** + `BufferedOutputStream(1MB)` + 1MB socket 接收缓冲，尽量快。
- 进度回调节流 200ms；顶栏/传输页显示已下载/总大小、速度、完成计数。
- 传输期间前台服务保活 + 唤醒锁，锁屏/切后台不中断。
- 失败/取消删除半成品文件，避免残留与重试重名副本。

**评估**：✅ 覆盖完整，性能已做过针对性优化（去分配、缓冲、socket 调优）。

**可优化**：
- 单通道串行是尼康 PTP/IP 的固有限制，无法并行多文件；若机型支持更大 PTP 数据分片可进一步提速（需真机抓包确认）。
- 通知是静态文案，未显示实时进度/数量（可做成带进度的通知）。

### 场景 4：Wi-Fi 掉线 / 相机断开
**用户预期**：掉线别把任务全标失败；重连后自动接着传。

**当前逻辑**：
- Wi-Fi 掉线 → `forceDisconnect()` 直接关 socket，立即中断阻塞中的下载读取。
- 掉线导致的失败（`IOException`）→ 当前任务放回 `WAITING`、**暂停队列**（不级联失败），剩余保持等待。
- 重连成功 → `onCameraConnected()` 复位残留 `TRANSFERING` → **自动续传**剩余队列。
- 用 `OutputWriteException` 区分"存储写入失败"与"相机掉线"，避免误判卡死队列。

**评估**：✅ 这是本轮重点加固，链路完整、边界清晰。

**可优化**：
- 续传是**整文件重传**（非断点续传）。PTP 标准无通用断点续传，尼康一般也不支持 `GetPartialObject` 到任意机型；维持整文件重传是稳妥选择。
- 暂停期间 UI 未显式提示"等待重连中"，任务停在 `WAITING`，可加一个轻提示。

### 场景 5：设置传输目录
**当前逻辑**：SAF `OpenDocumentTree` 选目录，持久化 URI 权限，显示可读路径。

**评估**：✅ 合理，SAF 是 Android 10+ 的正确做法。

**可优化**：可显示剩余空间；可选择"按日期建子目录"。

### 场景 6：重试失败任务
**当前逻辑**：单个任务或批量把 `FAILED/CANCELLED` 重置为 `WAITING` 再处理。

**评估**：✅ 合理。掉线已改为自动续传，此处主要面向"个别文件真失败"。

---

## 三、总体结论

当前软件**核心链路完整、健壮性经过多轮加固**，基本满足"连相机 Wi-Fi → 浏览 → 传输"的主诉求。**最大的体验缺口是文件列表没有缩略图**——这正是本次要补齐的功能。其余可优化项（筛选、剩余空间、通知进度、网段可配）属增强，不影响主流程。

## 四、缩略图最佳性能实现调研

### 4.1 缩略图从哪来？
关键点：**尼康相机通过 PTP `GetThumb (0x100A)` 直接提供缩略图**，不是从网络 URL 加载。所以 **Coil/Glide 这类基于 URL 的图片库用不上**——数据源是同一条 PTP/IP 连接返回的 JPEG 字节，需要我们自己取、解码、缓存。

- `GetThumb(handle)` 返回一张小 JPEG（尼康通常约 160×120），JPG/NEF/视频都能取到。
- 走的是**同一条命令通道**，与文件信息、下载**共用 `ioMutex`**——这决定了性能策略。

### 4.2 性能挑战与对策

| 挑战 | 对策 |
|---|---|
| **单通道串行**：缩略图请求会和下载/命令抢 `ioMutex` | 只加载**可见项**（`LazyVerticalGrid` 天然只组合可见单元）；传输进行时缩略图自然排在下载之后，不打断传输 |
| **重复请求**：滚动/重组会重复触发 | **内存 LRU 缓存**（`LruCache<handle, ImageBitmap>`），命中直接返回；`produceState(key=handle)` 同 handle 不重复启动 |
| **主线程卡顿**：解码 JPEG 是 CPU 活 | 取字节走 IO 线程，`BitmapFactory.decodeByteArray` 放 `Dispatchers.Default`，只在完成后回主线程 setState |
| **内存占用**：几百张缩略图 | LRU 按位图字节数限容（约 `maxMemory/8`），超限自动淘汰 |
| **换会话失效**：重连后 handle 可能变 | 每次 `loadFiles` 开始 `evictAll()` 清缓存 |

### 4.3 为什么用 `LazyVerticalGrid` + `produceState`
- `LazyVerticalGrid` 只组合并保留可见/邻近单元 → **只为可见缩略图发请求**，滚走即回收，天然限制并发量与内存。
- 每个单元用 `produceState(handle) { loadThumbnail(handle) }`：进入可见区触发一次加载，命中缓存则瞬时；离开可见区协程随之取消。
- 分组头用 `item(span = { GridItemSpan(maxLineSpan) })` **整行跨列**，实现"分组不变、仅展示形态变化"。

### 4.4 取舍与后续
- **本期**：内存 LRU + 懒加载 + 后台解码，足以支撑流畅滚动。
- **可选增强**（后续）：磁盘缓存（`DiskLruCache`）让缩略图跨重启/重连存活；缩略图预取（预加载下一屏）；传输进行时降低缩略图优先级（如给缩略图单独一个"礼让"标记，在有下载任务时暂缓）。

## 五、本次落地设计（缩略图模式）

### 5.1 设置项（持久化到 SharedPreferences）
- **缩略图模式**：开关（`thumbnailMode`，默认关）。
- **展示列数**：`thumbnailColumns`，可选 **1–4**（默认 3），仅在缩略图模式下生效。

### 5.2 展示规则
- **分组逻辑完全不变**（仍按日期分组、组内倒序），**只有展示形态不同**：
  - 关：现有文本列表。
  - 开：`LazyVerticalGrid`，分组头整行跨列，其下为 N 列缩略图方格。
- 缩略图单元：正方形裁剪缩略图 + 左上类型角标 + 右上/角落传输状态；点击=加入队列（未设目录同样引导到设置页）。
- 加载中显示占位（类型角标底色），解码失败回退占位。

### 5.3 数据流
```
FileListScreen(缩略图单元)
   └─ produceState(handle) ─▶ CameraViewModel.loadThumbnail(handle)
                                   ├─ LruCache 命中 → 直接返回
                                   └─ 未命中 → NikonCamera.getThumbnail(GetThumb, ioMutex)
                                                → Default 线程 decodeByteArray → 存缓存 → 返回
```

### 5.4 不影响既有能力
- 缩略图与下载共用 `ioMutex`，串行安全，不会与传输产生并发冲突；传输优先推进（下载持锁期间缩略图排队等待）。
- 关闭缩略图模式即完全回到原文本列表，零额外开销。
